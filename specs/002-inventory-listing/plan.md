# Implementation Plan: Inventory Page ‚Äî Product Listing

**Branch**: `002-inventory-listing` | **Date**: 2026-03-13 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-inventory-listing/spec.md`

## Summary

Implement end-to-end product listing test coverage for SauceDemo's inventory page
(`https://www.saucedemo.com/inventory.html`): product catalogue display (6 cards,
all 4 elements per card), four sort modes (A-Z, Z-A, low-high, high-low), product
detail navigation, and cart badge counter.

This feature **extends** the `InventoryPage` Page Object established in
`001-saucedemo-auth`, adding inventory-specific interactions (sorting, cart, detail
navigation) without replacing the auth-scoped foundation (goto, logout). A new
minimal `ProductDetailPage` is introduced for the back-navigation contract.

## Technical Context

**Language/Version**: TypeScript 5.x, `strict: true`, `noImplicitAny: true`
**Primary Dependencies**: `@playwright/test`, `allure-playwright`,
  `eslint`, `@typescript-eslint/eslint-plugin`

**Target Platform**: Web application ‚Äî `https://www.saucedemo.com`

**Performance Goals**: Full inventory suite (3 browsers) completes in < 60s on CI

  arrays (names) and numeric arrays (parsed prices); cart badge verified via
  numeric string comparison; `standard_user` only
**Scale/Scope**: 10 test scenarios, 1 test file, 2 Page Object classes
  (InventoryPage extended, ProductDetailPage new)

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| Principle IV: `data-test` instead of `data-testid` | SauceDemo (third-party) uses `data-test` as its test-attribute convention | Cannot add attributes to a site we don't control; semantically equivalent to `data-testid` |
| Principle IV: Product-specific `data-test` for add-to-cart buttons (`add-to-cart-{slug}`) | SauceDemo exposes no generic `add-to-cart` attribute; each button has a product-specific slug | No shared data-test value across all add-to-cart buttons; using index-based `nth()` inside `InventoryPage` is the most stable fallback while keeping tests selector-free |
| Principle IV: CSS class selectors for US1 structural counts (`.inventory_item`, `.inventory_item_desc`) | SauceDemo does not expose a stable `data-test` wrapper for product card containers; US1 requires structural parity assertions across product cards and descriptions | Counting only name elements does not prove full structural completeness; class-selector counts are confined to `InventoryPage` and keep test files selector-free |

## Phase 0: Research Summary

All 6 decisions documented in [research.md](research.md). No NEEDS CLARIFICATION
items remain.

Key decisions:
1. **InventoryPage extension strategy** ‚Äî extend existing class; new methods added
   without modifying the 001 auth-scoped interface
2. **SauceDemo inventory selectors** ‚Äî `data-test="product_sort_container"` (sort
   dropdown); `data-test="inventory-item-name"` (names); `data-test="inventory-item-price"` (prices); product-specific add-to-cart buttons; class-based structural selectors (`.inventory_item`, `.inventory_item_desc`) for US1 count parity
3. **Sort validation** ‚Äî collect all names/prices into arrays; compare to JS-sorted
   reference arrays (string `.sort()` for names; numeric `.sort((a,b) => a-b)` for prices)
4. **Price parsing** ‚Äî strip `$` prefix, `parseFloat()`; `toFixed(2)` comparison
   is not needed here (only ordering is asserted, not exact values)
5. **ProductDetailPage design** ‚Äî minimal two-method class: `isOnDetailPage()` +
   `goBack()`; URL pattern `/inventory-item.html?id=` confirms detail page
6. **Cart isolation** ‚Äî each test context loads fresh storageState (empty cart);
   cart additions do not bleed across tests
